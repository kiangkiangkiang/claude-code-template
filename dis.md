# CLAUDE.md
CLAUDE.md 內應該要有詳細的專案管理概念：
1. 善用使用者給的工具，使用者會把工具放在 .claude/tools 內，例如 .claude/tools/git 你可以查看 README.md 知道這個資料夾下工具的介紹
2. 善用 .claude/tools/git 工具做專案管控，你要善用裡面的內容，透過 git repo 管理在這個 codebase 內的所有 history
3. 抽象的指令，你需要有三個抽象指令概念：
   1. 系統指令，我等等要跟你討論，例如剛剛提到的 git 管控的開發流程，SDD 的開發流程，通常會在 .claude/docs 內
   2. 專案指令，這個 Github Repo 的目標，Epic 概念，通常是一個團隊或產品的核心價值與開發守則。通常也會放在 .claude/docs/ 內。透過是 .claude/docs/PROJECT.md 來描述
   3. 任務指令，一個專案，使用者會給你很多任務，你要依照上述兩個指令完成任務，這種指令通常不需要文件描述，通常會隨著 git 記錄在 issue 或是 commit 之類的。
4. SDD 開發模式，以上一點來說，你會有很多文件當作概念，全局的系統概念統一放在 .claude/docs 內，

所以 CLAUDE.md 要有以上的宏觀概念，並且透過文件指引的方式，指引 AI 去對應的位置查看文件系統，了解流程，CLAUDE.md 只會放這些指令的 Summary 以及宏觀的全局概念。

注意，PROJECT.md 是你要持續一直更新的（專案會進展，需求會被解決，會有新的需求，你要時時的紀錄大方向，所以這個檔案是會一直變動的，隨著你完成任務，隨著你了解專案，你看過各個 pr, issue, branch, git log，都會一直迭代 PROJECT.md）

# 開發流程

你會一直接收到使用者的任務，這些任務通常很抽象，你要根據使用者給的任務，拆成多個 issue (一個 issue 一個 task，issue 內容要列出這個 issue 想解決的問題)，共同解決這個任務，並且每次接到需求，都要更新 .claude/docs/TASK.md 表示當前需求（當然這個檔案也要被敘述在 CLAUDE.md 內），這個 TASK.md 就是記錄使用者當下的需求，以及你根據需求創建對應的 issue / branch，換句話說，每次啟動時，你都要看這個檔案的內容，因為這個檔案記錄著你的開發狀況，進度（當然可以看 git log 也行），隨時更新這個檔案，當需求解決，這個檔案理所當然要清空，有新的需求，就直接取代成新的需求即可

舉個例子，啟動時，你會先看 TASK.md github issue，看看有沒有上次沒進行完的需求，當使用者跟你講任務，你會判斷是否跟上次有相關，如果有，就會接續做，如果沒有，就會直接更新需求然後創建對應的 issue 創建 branch 開發，每次的進度都會記錄在 git commit 內，
